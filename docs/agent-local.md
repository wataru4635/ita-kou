## 知見: ビューポート端ブリードは 50vw でなく 50cqw（+ 親に container-type:inline-size）★恒久ルール

**ルール（ユーザー指定・kimpara恒久）**: 要素を画面端までブリードさせる
`margin-left/right: calc(50% - 50vw)` や `width: calc(50vw + …)` は、**スクロールバー幅分はみ出して
横スクロールが発生する**（`vw` はスクロールバーを含むビューポート幅のため）。

- **対策**: 一番外枠の section（例 `.order-content`）に `container-type: inline-size;` を付け、
  ブリードの `vw` を **`cqw`** に置き換える（`calc(50% - 50cqw)` / `width: calc(50cqw + …)`）。
  `cqw` はスクロールバーを除いたコンテナ幅基準なので横スクロールが出ない。
- container-type を付けた要素自身のプロパティ内の `cqw` は祖先コンテナを参照する（自己循環しない）ため、
  ブリードする box 自身に container-type があっても問題ない。
- **今後ビューポート端ブリードを書くときは必ずこの方式（cqw + 外枠 container-type:inline-size）を使う**。
  `calc(50% - 50vw)` を新規に書かない。
- **派生注意（リキッドrem環境）**: kimpara は base がリキッド（rem が vw連動）なので、`width: rem(X)` の固定幅も
  **狭PC幅（768付近＝mq境界）で拡大して横はみ出しの一因**になる（実例: 千鳥写真 `width: rem(390)` が
  768で約769pxに膨らみ +1px 横スクロール）。固定幅の写真等は **`width: min(#{rem(X)}, Ycqw)`** で上限を付ける
  （Y ≒ X ÷ 1366 × 100 をわずかに上回る値。例: `rem(390)`→`29cqw`、`rem(376)`→`28cqw`）。
  これで広い画面では rem（デザイン値）、狭PC幅では cqw が頭打ちになり横スクロールが出ない。

## 落とし穴: ita-kou の root font-size は 768〜1200px で「縮小」する（rem は自動追従。myClamp/px固定で再現しない）

`_vite/src/sass/base/_base.scss`（sp-first）の実挙動（実測）:
- 375〜767px: `1rem = 16px`（SP 固定）
- **768〜1199px**: `font-size: vw(1200,16)` = `1.3333vw` → **768px で 1rem ≒ 10.24px、1200px で 16px**（線形に縮小→拡大）
- 1200px 以上（`$inner=1200`）: `1rem = 16px` 固定（デザイン実寸）

**実例（ヘッダー PC ナビ）**: ナビバー合計はデザイン実寸で約 772px。`@include mq("md")`(768) でナビを出す際、
「1rem=16px 前提」で計算して「768px では 772px が収まらない→横スクロール」と誤判断し、`myClamp(92,133.5,768,1440)`（px固定 clamp）でセル幅だけ固定してしまった。
結果、セルは 92px 固定なのに `width: rem(104)` のお問い合わせセルだけ 66.5px に縮小（rem 追従）し、**比率が崩れた**。

**対策**:
1. このプロジェクトは**リキッド**。幅・余白・フォントは **素の `rem()` で書けば 768〜1200 で自動的に比例縮小**し、1200px 以上でデザイン実寸になる。`myClamp`/px固定で「収める」処理は不要（むしろ比率を壊す）。
2. `@include mq("md")` で出す PC レイアウトでも、横スクロール検証は **rem が縮小する前提**で行う（768px では実寸 ×0.64）。例: ナビ 772px は 768px で約 494px に縮小して収まる。
3. ロゴ等の SP→PC でサイズが変わる要素も、`rem()` 同士なら境界（767→768）で `rem(211)@16px ≒ rem(331)@10.24px` のように**滑らかに繋がる**。
4. **作業前に必ず `base/_base.scss` の root font-size 挙動を確認**してから「収まる/はみ出す」を判断する。

## 規約: calc() 内の rem() に `#{}`（補間）は付けない

Dart Sass（Vite）は `calc()` 内でもユーザー定義 `rem()` を評価するため、補間は不要。
- ✅ `right: calc(50% + rem(120));` → `calc(50% + 7.5rem)` と出力（実機検証済み。rem(121)→7.5625rem も確認）
- ❌ `right: calc(50% + #{rem(120)});`（冗長）
- `rem` は CSS の剰余関数名と同名だが、Dart Sass はユーザー定義関数を優先評価するので衝突しない。
- `#{}` が必要なのは**文字列補間**のみ（例: `_breakpoints.scss` の `"(min-width: #{map.get(...)}px)"` や `@media #{...}`）。`calc()` の数値計算では付けない。
